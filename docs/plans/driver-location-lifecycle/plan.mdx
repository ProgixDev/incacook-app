### Driver App — Location and Verified Delivery Lifecycle

The driver app now derives one GPS mode from two facts: whether the driver is
online and whether they own an active delivery. A single
`DriverLocationModeCoordinator` observes those facts and applies `off`,
`foreground`, or `background` through `LocationService.applyMode`. Delivery,
incoming-order, and route controllers mutate domain state but no longer start or
stop the shared location stream themselves.

The backend enforces the matching and handoff boundaries used by this client:
only `READY` orders can be listed or claimed, no-driver timeout atomically retires
the searchable delivery, and QR/absent delivery completion remains retryable if
order or wallet finalization fails.

The resolver is deliberately defensive: `(offline, active job)` resolves to
`off`. Under normal server invariants a driver cannot claim while offline, but a
single defined result prevents an ambiguous half-tracking state during restore
or cancellation races.

<Callout id="completion-order" tone="decision">

Order/wallet completion is idempotent and runs before the Delivery row is marked
`DELIVERED`. If finalization fails, the Delivery remains in an out-for-delivery
state and the driver can retry. If the subsequent Delivery update fails, retrying
is also safe because the order operation is idempotent.

</Callout>

### Scope boundary

This implementation does not change wire formats or database schema. Driver
statistics and hydrated incoming-order filler remain separate gaps in feature
spec 008. Production multi-device and background-location observation remains a
manual QA responsibility because unit tests and successful builds cannot prove
OS suspension behavior.
